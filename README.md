# 1. В чём разница между debug, profile и release сборками?
## Debug
Используется во время разработки.
Включает отладочную информацию, hot reload, подробные логи и проверки.
Работает медленнее и не подходит для публикации.
## Profile
Предназначена для анализа производительности приложения.
Отключена большая часть отладки, но сохраняются инструменты профилирования.
Используется для поиска узких мест перед релизом.
## Release
Финальная версия приложения для пользователей.
Код оптимизирован, ресурсы сжаты, отладка отключена.
Требует подписи и используется для публикации в Google Play.

# 2. Зачем нужно подписывать приложение и что содержит файл .jks?
Подпись приложения необходима для:
подтверждения авторства приложения;
защиты от подмены и модификации;
возможности обновления приложения в магазине.
Файл .jks (Java KeyStore) содержит:
криптографический ключ;
сертификат разработчика;
пароль хранилища и ключа.
Этот файл используется для подписи release-сборок Android-приложений.

# 3. Чем отличается APK от AAB и что предпочтительнее для публикации?
APK
Готовый установочный файл, содержащий все ресурсы приложения.
Подходит для ручной установки и тестирования.
AAB (Android App Bundle)
Формат публикации, из которого Google Play сам собирает оптимальные APK под конкретное устройство.
Позволяет уменьшить размер приложения и оптимизировать доставку.
Для публикации в Google Play предпочтителен AAB, так как это современный и рекомендуемый формат.

# 4. Какие обязательные документы требуются для размещения приложения в Google Play?
## Для публикации приложения в Google Play разработчику необходимо предоставить:
Политику конфиденциальности (Privacy Policy), если приложение собирает персональные данные;
Описание приложения (краткое и полное);
Иконку приложения и графические материалы (скриншоты, при необходимости — feature graphic);
Контактные данные разработчика (email обязательно);
Заполненную информацию о контенте и возрастных ограничениях.

# 5. Как проверить, что приложение не содержит debug-логов и лишних пакетов?
## Проверка выполняется следующими способами:
Сборка приложения в режиме release (flutter build appbundle --release);
Отсутствие вызовов print(), debugPrint() и assert() в коде;
Проверка, что в Gradle отключена отладка и включена оптимизация (minifyEnabled, shrinkResources);
Анализ содержимого AAB/APK с помощью Android Studio → Analyze APK.

#6 Какие типичные ошибки совершают разработчики при первой публикации?
## Наиболее распространённые ошибки:
Несовпадение package name с зарегистрированным в Firebase или Google Play;
Отсутствие политики конфиденциальности;
Попытка загрузить debug-сборку вместо release;
Неправильно настроенная подпись приложения;
Забытые тестовые данные или заглушки в интерфейсе;
Несоблюдение требований Google Play к контенту и описанию приложения.
